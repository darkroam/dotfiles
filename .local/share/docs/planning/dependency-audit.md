# 依赖与文档审计计划

状态：已制定。本轮仅面向 Debian 审计并确保其可运行；其他发行版的兼容性不作为
本轮通过条件。在审阅并记录上一阶段结果前，不进入下一阶段。未经明确决定，不得
安装、删除或修改任何依赖。

## 范围与规则

- 仅审计已跟踪文件；未跟踪的机器专属状态不在范围内。
- 安装状态、软件包名称、服务提供者和运行行为均以当前 Debian 系统为准。保留
  配置中的跨发行版分支，但不在本轮验证 APT 以外的包管理器或发行版；涉及其他
  发行版的代码仅做语法检查。
- 依赖包括被跟踪配置所需的外部命令、软件包管理器、服务、可执行程序、运行时、
  字体、主题、数据提供者或系统设施。
- 必需命令缺失时，记录其 Debian 软件包名称并报告，然后暂停该项，直到确认完成
  安装。
- `docs/project/dependencies.md` 是唯一权威的依赖清单；已迁移的
  `larbs/progs.csv` 内容保留在其 LARBS 迁移参考中。
- 向 `dependencies.md` 添加信息前，先选择职责相符的既有 layout。仅当正在使用且
  具有连贯职责的功能领域没有合适归属时才增加 layout；不得为单个边缘工具建立
  过细分类。
- 过时或发行版专属的 LARBS 条目应作为历史参考记录，不得写成当前必需依赖。
- 标记为最低优先级可选的功能不阻塞本轮 Debian 审计；仅验证代码完备性，包括
  调用路径、条件处理和缺失时的失败行为，记录其用途与排除理由，但不要求安装或
  运行验证。
- 始终以当前 Debian 配置可用为前提；修改不得破坏已验证的启动、shell、脚本或
  文档链接路径。
- 文档必须一致且完整。依赖名称、layout、要求级别、状态或行为发生变化时，同步
  核对根 README、依赖清单、架构文档、用户指南和规划记录。

## 审计记录

每个已审查依赖均维护以下字段：

| 字段 | 含义 |
| --- | --- |
| 来源文件 | 需要或说明该依赖的已跟踪文件 |
| 依赖 | 命令、服务、软件包、运行时或系统设施 |
| 要求级别 | 必需、可选、回退、仅模板或历史 |
| 安装状态 | 已安装、缺失或当前环境无法验证 |
| dependencies layout | `dependencies.md` 中既有或新批准的分类 |
| `progs.csv` 状态 | 不存在、待迁移、已迁移、重复、历史或已删除 |
| 处理结果 | 通过、补充文档、等待安装、挂起或讨论 |

## 第一阶段：逐文件依赖审计

状态：已完成。所有已跟踪运行配置、脚本、桌面入口、数据资源与兼容链接均已有记录
结果或明确挂起原因；模板和文档内容按后续阶段处理。

1. 分批列举已跟踪的配置、脚本、模板、桌面条目和文档。
2. 检查每个文件中的依赖引用。
3. 无依赖文件标记为通过。
4. 对每个发现的依赖检查本机安装状态。
5. 若未安装，记录该项并停止验证，直到确认安装完成。
6. 对已安装依赖，定位其当前 `dependencies.md` layout 及对应的 `progs.csv` 行；
   此时不删除 `progs.csv` 数据。

退出条件：每个已跟踪文件都有记录结果或明确的暂停原因。

## 第二阶段：完整迁移 `progs.csv`

状态：已完成。全部 81 行已归类并迁移；`progs.csv` 仅保留格式头，原始包名、用途与
处置结论保留在 `dependencies.md` 的 LARBS 迁移参考中。

1. 处理 `larbs/progs.csv` 的每一行，包括第一阶段未发现的条目。
2. 编辑前将每一行归类为：
   - 当前活跃依赖；
   - 当前可选依赖；
   - 回退或仅模板依赖；
   - 历史 LARBS 参考；
   - 重复或已被替代的条目。
3. 将有价值的用途说明、替代方案、来源背景和限制迁入相应 layout 的
   `dependencies.md`。
4. 先评估既有 layout。只有持续存在、职责连贯的活跃能力才新增 layout；其他
   情况归入最接近分类并清楚标注专用用途。
5. 仅当信息已迁移，或其不适用原因已在 `dependencies.md` 明确记录后，才删除
   对应 `progs.csv` 行。

退出条件：每个 `progs.csv` 行均有明确处置，且没有未解释的活跃依赖只存在于该文件。

## 第三阶段：依赖清单完整性

状态：已完成。已反向核对当前运行配置，移除无引用的活跃必需项，标注条件回退、
外部源码参考和 Debian 挂起项，并保持编译工具链的既定待讨论范围。

1. 在修改用户和架构文档前，先审阅最终的 `dependencies.md` layout 与列结构。
2. 只添加能提升维护性的字段，例如配置引用、要求级别、安装状态、替代方案和来源
   备注。
3. 对每一项已记录依赖，搜索已跟踪配置和脚本中的实际引用。
4. 将无匹配条目归类为基础环境、历史、仅模板，或待讨论的删除候选。
5. 补齐缺失的用途、引用、替代关系和状态信息。

退出条件：清单内部分类一致，每个活跃依赖均有来源说明，每个 layout 都有清晰边界。

## 第四阶段：文档对齐

状态：已完成。用户指南与架构文档均按最终十个 layout 增加独立章节；原有工作流、
启动说明、上游差异和维护边界保留。

1. 使用最终 `dependencies.md` layout 作为唯一分类模型。
2. 重组 `docs/user/desktop-guide-zh.md`，为每个 layout 建立章节，说明功能、常用
   入口、配置位置及可选/回退行为。
3. 重组 `docs/project/architecture.md`，为每个 layout 建立章节，说明归属、文件、
   运行时关系和依赖边界。
4. 根 `README.md` 继续作为安装和导航入口，不重复详细内容。

退出条件：依赖 layout、用户指南和架构文档使用相同名称，并描述一致行为。

## 验证与提交纪律

- 每个修改批次后执行相关语法检查、Markdown 链接检查、`git diff --check` 和暂存
  文件审阅。
- 每个批次完成后，复核受影响文档之间的术语、layout、链接、状态和结论是否一致，
  并确认没有遗漏必须说明的用户可见行为。
- 删除旧 `progs.csv` 行前，审阅实际差异。
- 修改积累到可独立验证的检查点时，先完成 Debian 验证、语法检查和差异审阅，
  再请求提交；不要让未经验证的改动长期堆积。
- 仅在明确批准后提交一个连贯且已审阅的批次，以保留可工作的回退点。
