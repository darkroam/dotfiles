# 跨发行版可用性、依赖与文档审计流程

本文件是本配置库的可复用审计流程。目标不是让所有发行版使用相同的软件包名，而是
验证已跟踪配置在目标发行版上具备可运行的代码路径、可映射的依赖，以及完整且互相
关联的文档。

## 当前基线

- Debian 首轮审计已完成：运行配置、脚本、桌面入口、依赖清单、用户指南和架构文档
  已完成对应检查。
- `.local/share/docs/project/dependencies.md` 是依赖能力、layout、替代关系和历史迁移
  说明的唯一权威来源。
- `.local/share/docs/planning/dependency-audit-record.md` 保存已执行审计的事实记录。后续
  发行版在同一文件中按发行版、版本和日期新增独立记录段落。
- `larbs/progs.csv` 的 81 条旧 LARBS 数据已迁移至 `dependencies.md` 的 LARBS 迁移
  参考。该文件只保留格式头，**后续任何发行版审计都不再处理它**。

## 总体规则

- 仅检查已跟踪文件。设备状态、凭据、缓存、构建产物和 `*.local` 个人覆盖文件不在
  审计范围内，也不得因审计而读取或提交。
- 每次审计先记录目标发行版、版本、架构、会话类型和包管理器。软件包名、服务提供
  者和默认路径必须以目标系统事实为准，不从 Debian 或其他发行版直接推断。
- 依赖按能力记录，软件包名按发行版映射。例如记录“MPD 控制客户端 `mpc`”，再记录
  目标发行版提供它的软件包；不要把一个发行版的包名写成通用要求。
- 保留跨发行版分支。当前目标以外的分支至少做对应解释器的语法检查；不因本机未安装
  另一发行版包管理器而删除可信配置。
- 缺失的必需依赖、错误的服务假设或无法到达的代码路径必须记录。安装、替换、降级为
  可选或挂起，都要先说明理由并在记录中留下结论。
- 最低优先级可选项只验证调用保护、缺失行为和文档说明，不要求安装或运行。条件回退
  必须保证其主会话、状态栏或 shell 不会因缺失命令失败。
- 任何改动都必须保持已验证的启动、shell、脚本和文档链接可用；修改积累到独立检查
  点时提交，避免未验证变更长期堆积。

## 每次发行版审计流程

### 0. 建立目标基线

1. 记录发行版、版本、架构、内核、会话类型、包管理器和可用 init/service 管理方式。
2. 记录当前提交、工作树状态和未跟踪文件清单；未跟踪文件只用于排除，不读取其内容。
3. 确认根兼容链接、`c` 命令、XDG 路径和 `~/.local/bin` 的加载方式符合目标 shell。
4. 在审计记录中创建本次运行段落，明确哪些运行验证受容器、无图形会话或无用户 D-Bus
   限制而只能静态检查。

### 1. 代码与配置静态有效性

1. 按解释器检查所有已跟踪脚本：POSIX Shell 用 `sh -n`，Bash 用 `bash -n`，Zsh 用
   `zsh -n`，Python 用编译检查；配置格式按可用工具做解析或最小加载检查。
2. 对目标发行版分支验证实际命令、参数和服务名称；其他发行版分支只验证语法和条件
   保护。
3. 检查桌面入口的 `Exec` 路径、兼容链接、模板路径、文件权限和引用的本地脚本。
4. 对关键启动链执行非破坏性检查：shell 加载、X11 配置语法、systemd 用户服务定义
   或等价服务入口。不能在当前环境运行的图形/用户服务路径必须标为“会话复查”，不能
   伪造通过。

退出条件：每个已跟踪运行文件都有“通过、条件回退、会话复查、挂起或待修复”之一的
明确状态。

### 2. 依赖能力与发行版映射

1. 从已跟踪配置、脚本、桌面入口和服务定义提取外部命令、运行时、字体、主题、网络
   服务和硬件接口。
2. 在 `dependencies.md` 的现有 layout 中定位每项能力；只有持续、职责连贯的新领域
   才新增 layout，不能为单个工具建立孤立分类。
3. 对每项能力确认目标发行版的提供者：包名、源码构建、服务、内核功能或外部安装方式。
   记录命令实际可执行、包已安装、仅条件可选或目标发行版无可用提供者。
4. 反向审查 `dependencies.md`：无运行引用的条目必须标为基础环境、历史、外部源码、
   仅模板或可选项；不得保留成“新机必装”的模糊要求。
5. 发现缺失必需依赖时暂停受影响项目，报告目标发行版的准确安装方式；安装后重新验证
   命令、脚本语法和受影响路径。

退出条件：每个活跃依赖都有来源、layout、目标发行版提供者和要求级别；每个无引用条目
都有明确归类。

### 3. 运行路径与服务验证

1. 在目标用户的真实会话中验证登录 shell、`startx`/显示管理器、输入法、Xresources、
   壁纸回退、通知、音频服务和状态栏。
2. 验证按需入口：文件预览、桌面 MIME、下载队列、Transmission、挂载、截图、录制、
   OTP、网络管理和媒体控制。没有硬件、网络或账户条件的项目记录为会话复查，不强行
   伪造测试环境。
3. 验证目标发行版的服务归属不会冲突，例如 PipeWire 的 systemd 用户服务不得与 X11
   启动脚本重复拉起。
4. 对可选和回退项确认缺失时的行为仍符合文档：静默隐藏、明确提示或可用替代路径。

退出条件：关键启动与交互路径可运行，或每个未验证路径都有可复现的环境限制和后续
测试条件。

### 4. 文档关联性与完整性

1. 以 `dependencies.md` 的 layout 为唯一分类模型，检查
   `architecture.md` 和 `desktop-guide-zh.md` 是否各自覆盖每个 layout，且描述的入口、
   所有权、可选/回退和服务行为一致。
2. 检查根 `README.md` 仍是安装、启动逻辑、感谢/版权和文档导航入口；链接必须指向
   当前文件，不能复制出冲突的依赖清单。
3. 检查 planning 文档：`todo.md` 只保留活动工作，`suspended.md` 只保留有恢复条件的
   挂起工作，`history.md` 只记录已完成结论，审计记录保留可追溯事实。
4. 对新增、删除、替换或降级的依赖，同步核对 README、依赖清单、架构、用户指南、
   审计记录和相关待办；术语、layout、状态、链接和用户可见行为必须一致。
5. 检查 Markdown 链接、相对路径、桌面文件引用和文档中的命令示例。无法验证的外部
   网络链接要标明其性质，不把网络可达性误写成配置成功。

退出条件：文档之间没有相互矛盾的依赖、入口、状态或链接；每个 layout 在用户和维护
文档中都有可定位的说明。

### 5. 复查与提交

1. 执行受影响解释器的语法检查、`git diff --check`、链接/路径检查和目标发行版的最小
   命令验证。
2. 审阅 staged diff，确认只包含已跟踪且属于本次审计的文件；绝不提交日志、缓存、
   凭据、设备状态或其他未跟踪文件。
3. 在提交前复核记录、依赖清单、架构和用户指南的术语与结论；必要时先修正文档再
   提交。
4. 以一个可独立验证的提交结束本次审计。提交信息应包含目标发行版或审计范围。

## 一次性历史迁移

`progs.csv` 的迁移已经完成，不是本流程的一部分。只有在未来重新引入新的、明确来源
的历史清单时，才需要另行建立迁移计划；不得把空的 `progs.csv` 当作后续审计待办。
